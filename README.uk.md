# DTEK Telegram Bot

[![Відкрити Home Assistant і додати цей репозиторій у магазин add-on.](https://my.home-assistant.io/badges/supervisor_add_addon_repository.svg)](https://my.home-assistant.io/redirect/supervisor_add_addon_repository/?repository_url=https%3A%2F%2Fgithub.com%2Fgroove-max%2Fha-dtek-telegram-bot)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)

[English README](README.md)

Home Assistant add-on для Telegram-сповіщень про наявність або відсутність електроенергії, зміну статусів від ДТЕК, планові графіки, якість напруги та живе закріплене статусне повідомлення.

Add-on створений для спільної роботи з [ha-dtek-monitor](https://github.com/groove-max/ha-dtek-monitor). Інтеграція надає дані ДТЕК по конкретній адресі, а add-on поєднує ці сутності з локальними сенсорами Home Assistant, наприклад сенсорами напруги, частоти або фаз, щоб отримувати швидші й точніші сповіщення.

Найкраще він підходить для чатів багатоквартирних будинків, локальних районних чатів та інших сценаріїв, де одна адреса має один спільний Telegram-канал або групу для інформування мешканців.

## Важливо про встановлення

Цей проєкт є **Home Assistant add-on**, а не HACS-інтеграцією.

- **Цей add-on** встановлюється через **Add-on Store** як **custom add-on repository**.
- **ha-dtek-monitor** встановлюється через **HACS** або вручну як кастомна інтеграція.

HACS не встановлює Home Assistant add-on.

## Навіщо цей add-on потрібен

Офіційні дані ДТЕК корисні, але для оперативних сповіщень їх часто недостатньо:

- ДТЕК може показувати відключення із запізненням відносно реального стану.
- ДТЕК може вже повернути `ok`, хоча світла фактично ще немає.
- Планові графіки можуть існувати окремо від реального поточного стану.
- Локальні сенсори у Home Assistant можуть краще знати, що реально відбувається з живленням.

Тому add-on поєднує два джерела:

- `ha-dtek-monitor` для офіційного статусу, причин відключення, графіків і груп
- Локальні сенсори Home Assistant для валідації фактичної наявності світла
- Telegram-логіку доставки, зручну для одного бота в одному груповому чаті

## Основні можливості

- Telegram-сповіщення про початок, завершення, зміну причини та зміну ETA відключень за даними ДТЕК
- Сповіщення про зміну графіка та попередження про найближче планове відключення
- Визначення електропостачання за ДТЕК, за локальною напругою або за швидким тригером втрати плюс підтвердження напругою
- Однофазна та багатофазна логіка з трьома станами будинку: `on / partial / off`
- Окремі сповіщення про низьку, високу та нормалізовану напругу
- Окремі сповіщення по фазах для часткових відключень
- Живе закріплене статусне повідомлення з періодичним оновленням
- Вбудований ingress UI для налаштування, діагностики, редагування шаблонів, прев’ю та імпорту/експорту
- Імпорт/експорт окремої адреси або всього конфігу add-on між різними інстансами Home Assistant

- Працююча інтеграція [ha-dtek-monitor](https://github.com/groove-max/ha-dtek-monitor) для тієї ж адреси

- Налаштовувати окрему адресу на кожен будинок або під’їзд, якщо локальна електрична топологія відрізняється

## Які сутності читає add-on

Add-on розрахований на сутності, які створює `ha-dtek-monitor`, зокрема:

- `binary_sensor.<address_slug>_power`
- `sensor.<address_slug>_outage_status`
- `sensor.<address_slug>_outage_description`
- `sensor.<address_slug>_outage_start`
- `sensor.<address_slug>_outage_end`
- `sensor.<address_slug>_schedule_group`
- `sensor.<address_slug>_schedule_changed`
- `calendar.<address_slug>_outage_schedule`

Додатково можна підключити власні локальні сенсори:

- один сенсор напруги для однофазної схеми
- два або три сенсори напруги для багатофазної схеми
- швидкий сенсор втрати живлення, наприклад частоту AC від інвертора чи лічильника

## Встановлення

### 1. Спочатку встановіть DTEK Monitor

Якщо це ще не зроблено:

1. Встановіть `ha-dtek-monitor`.
2. Додайте потрібну адресу.
3. Переконайтесь, що сутності ДТЕК з’явилися у Home Assistant.

### 2. Додайте цей репозиторій у Add-on Store

1. Відкрийте `Settings -> Add-ons -> Add-on Store`.
2. Відкрийте меню у правому верхньому куті.
3. Оберіть `Repositories`.
4. Додайте:

   `https://github.com/groove-max/ha-dtek-telegram-bot`

5. Закрийте діалог.

### 3. Встановіть add-on

1. Відкрийте `DTEK Telegram Bot`.
2. Натисніть `Install`.
3. Дочекайтесь завершення збірки.

### 4. Заповніть базові параметри

Мінімально потрібно вказати:

- `telegram_bot_token`
- `telegram_chat_id`

Після цього запустіть add-on.

### 5. Відкрийте UI add-on

Після старту відкрийте ingress-інтерфейс із сайдбару або зі сторінки add-on.

## Перше налаштування

Основне налаштування виконується у вбудованому ingress UI.

Типовий перший сценарій:

1. Відкрити вкладку `Configure`.
2. Додати адресу з discovery.
3. Вибрати `entity_prefix`, який відповідає адресі з `ha-dtek-monitor`.
4. Вказати назву адреси для повідомлень у Telegram.
5. Обрати режим визначення живлення.
6. Додати сенсори напруги, якщо потрібна локальна валідація.
7. Увімкнути потрібні типи сповіщень.
8. Натиснути `Validate`.
9. Зберегти конфіг.
10. За потреби перезапустити add-on.

## Як працює визначення наявності світла

Add-on вміє трактувати електропостачання не лише як простий `on/off`.

### `dtek_only`

Використовується лише сутність живлення від ДТЕК.

Підходить, якщо:

- у вас немає локальних сенсорів напруги
- потрібна максимально проста схема

Компроміс:

- ДТЕК може відставати від реального стану

### `voltage_only`

Використовується лише локальна топологія напруги.

Підходить, якщо:

- ви більше довіряєте локальним сенсорам
- потрібно, щоб стан будинку відображав фізичну ситуацію

### `loss_plus_voltage`

Один швидкий сенсор втрати використовується як ранній тригер, а остаточне підтвердження відключення і відновлення робиться за сенсорами напруги.

Підходить, якщо:

- є сенсор, який миттєво реагує на втрату мережі
- сенсор напруги може переставати оновлюватися після відключення
- потрібна швидка реакція без довіри до застарілого останнього значення

## Однофазна та багатофазна логіка

Секція `Напруга та фази` є спільним джерелом правди для всієї voltage-aware логіки.

### Однофазна схема

Якщо задано один сенсор напруги:

- будинок має лише стани `on` або `off`
- фазні сповіщення практично не потрібні
- у статусі зазвичай показується `L1`

### Багатофазна схема

Якщо задано два або три сенсори напруги:

- `on` означає, що всі налаштовані фази присутні
- `partial` означає, що частина фаз відсутня, але не всі
- `off` означає, що відсутні всі фази

Це зручно для будинків із трифазним вводом або розділеною локальною мережею.

## Як трактуються unavailable значення

Сенсори напруги поводяться по-різному при втраті живлення:

- одні повертають `0`
- інші повертають `unavailable`
- деякі певний час зберігають останнє значення

Add-on дозволяє явно керувати цим через опцію `Unavailable = відсутня фаза`.

- Якщо опція увімкнена, `unavailable` може впливати на `partial` або `off`
- Якщо вимкнена, `unavailable` вважається невизначеним станом і не означає автоматичну втрату живлення

У режимі `loss_plus_voltage` add-on також захищається від застарілих voltage samples, очікуючи свіжий апдейт під час confirm window.

## Типи сповіщень

### Відключення за даними ДТЕК

Ці повідомлення будуються з даних `ha-dtek-monitor`:

- початок відключення
- завершення відключення
- зміна причини
- зміна орієнтовного часу відновлення

Якщо ДТЕК змінює кілька параметрів одразу, add-on намагається об’єднати це в одне повідомлення.

### Зміна графіка

Коли ДТЕК змінює графік, add-on може надіслати оновлений перелік слотів або повідомити, що планові відключення більше не очікуються.

### Зміна групи графіка

Якщо для адреси змінилася група, add-on може повідомити нову групу та поточний графік для неї.

### Попередження про найближче планове відключення

Add-on може заздалегідь попереджати про найближче планове вікно відключення.

Можна вибрати, коли воно має спрацьовувати:

- завжди
- лише коли світло є
- лише коли світла немає

### Сповіщення про напругу

Окремі повідомлення про напругу є опціональними.

Вони корисні, якщо:

- електропостачання формально є
- але якість напруги погана і техніка може працювати нестабільно

Секція напруги також підтримує гістерезис для повідомлень про нормалізацію.

- Low/high alert, як і раніше, спрацьовує на заданому порозі.
- Повідомлення `voltage_normal` надсилається тільки після повернення значення в безпечну зону із заданим запасом.
- Це зменшує спам, коли напруга коливається біля межі порога.

### Повідомлення про втрату та відновлення світла

Це основні повідомлення для більшості користувачів.

Вони можуть працювати за:

- даними ДТЕК
- локальною напругою
- loss trigger плюс voltage confirmation

У багатофазній схемі add-on також може надсилати окремі повідомлення про втрату та відновлення конкретної фази.

### Статусне повідомлення

Статусне повідомлення орієнтоване на групові чати.

Воно може:

- редагувати одне закріплене повідомлення
- або надсилати нові статусні повідомлення

У статусі відображаються:

- поточний стан живлення
- поточний стан фаз
- остання причина відключення та ETA, якщо вони доступні
- найближче планове відключення, якщо воно є
- час останнього оновлення

## Шаблони повідомлень

Усі Telegram-повідомлення побудовані на шаблонах.

Через UI можна:

- переглядати прев’ю будь-якого шаблону
- редагувати й зберігати власну версію
- повертати шаблон до стандартного стану
- надсилати тестове повідомлення

Це зручно для різних типів чатів:

- чат мешканців будинку
- районний чат
- власні формулювання причин відключення або статусів

## Імпорт та експорт

Add-on підтримує два рівні перенесення конфігурації:

### Імпорт/експорт однієї адреси

Підходить, коли потрібно перенести лише одну адресу з одного Home Assistant на інший.

### Імпорт/експорт усього конфігу

Підходить, коли треба перенести весь draft-конфіг add-on на інший інстанс.

Експортований JSON можна одразу завантажити у браузері. Він не зобов’язаний зберігатися всередині add-on на диску.

Важливо:

- на цільовому Home Assistant уже повинні існувати відповідні сутності
- add-on не створює сутності `ha-dtek-monitor` сам по собі

## Рекомендовані сценарії

### Чат багатоквартирного будинку

Рекомендовано:

- одна адреса
- один спільний груповий чат
- режим `loss_plus_voltage`
- закріплене статусне повідомлення
- увімкнені DTEK outage updates
- увімкнені попередження про найближчі планові відключення

### Приватний будинок з одним надійним сенсором напруги

Рекомендовано:

- одна адреса
- один однофазний сенсор напруги
- `voltage_only` або `loss_plus_voltage`
- окремі low/high voltage alerts, якщо техніка чутлива

### Трифазний будинок

Рекомендовано:

- два або три сенсори напруги з мітками `L1`, `L2`, `L3`
- `loss_plus_voltage`
- увімкнені фазні сповіщення
- увімкнене статусне повідомлення, щоб було видно `partial`

## Типові проблеми

### Add-on не надсилає повідомлення в Telegram

Перевірте:

- токен бота валідний
- chat ID правильний
- бот доданий до групи або каналу
- у логах немає Telegram rate-limit або permission errors

### Немає сутностей DTEK Monitor

Цей add-on залежить від `ha-dtek-monitor`.

Якщо його сутності відсутні, add-on не зможе побудувати DTEK-aware сповіщення для цієї адреси.

### Неправильно визначається наявність світла

Перевірте:

- вибраний режим визначення живлення
- список сенсорів напруги
- `present_above`
- `Unavailable = відсутня фаза`
- `loss_entity` і `loss_state`

Після цього порівняйте поточні значення у вкладці `Diagnostics`.

### Закріплене статусне повідомлення подвоїлося

Зазвичай це означає, що старе повідомлення не вдалося відредагувати через Telegram API.

Краще використовувати окремий bot token для цього add-on і не ділити його з іншими системами, які часто постять або редагують повідомлення.

### ДТЕК показує, що відключення немає, а світла фактично немає

Це реальний сценарій, який add-on вміє відобразити.

У такому випадку статус може показувати:

- `power_state = off`
- локальну тривалість відсутності світла
- `reason: unknown`
- окрему примітку, що ДТЕК не фіксує активного відключення

## Обмеження

- Add-on не може покращити самі дані ДТЕК, він лише поєднує їх із локальними сигналами Home Assistant.
- Якщо локальних сенсорів немає, затримки ДТЕК неминучі.
- Якщо локальні сенсори нестабільні, неправильний вибір topology може давати помилкові спрацювання.
- Доставка в Telegram усе одно залежить від доступності API Telegram і його rate limits.
